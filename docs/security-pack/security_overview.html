Data Protection at a Glance

Whistle Connect connects grassroots coaches with qualified, FA-registered referees — including young officials. We are the data controller for the personal data we hold and we process it under the UK General Data Protection Regulation (UK GDPR) and the Data Protection Act 2018.

The principles we build to
  • Data minimisation — we collect and share only what a task needs, nothing more.
  • Least privilege — access is enforced at the database, scoped to each user and role.
  • Safeguarding first — protections for under-18 referees are enforced in code, not just policy.
  • Security in transit — all traffic is encrypted over HTTPS; payments never touch our servers.
This document explains, in plain English, what we collect, how we verify referees with the FA, how we keep your money and messages safe, and how you can delete your data at any time.

Security & Data Protection — Access Control
04
Identity, Authentication & Access

Who can see what is decided by the database, not just the screen. Even a mistake in the interface cannot reveal one user’s data to another, because access rules are enforced one layer deeper.

Secure Sign-In

Authentication is managed by Supabase Auth with encrypted, cookie-based sessions and automatic token validation.

Row-Level Security

Every application table enforces per-user access policies in the database, so users only ever read and write their own rows.

Role-Based Access

Coach, referee and admin each see a scoped view. Admin tools are gated to verified administrators only.

Hardened Functions

Privileged database functions run with a pinned, safe search path and have anonymous execution revoked.

Least-Privilege Service Role

The all-access service key is used only by trusted system jobs (cron, webhooks), never exposed to the browser.

Encrypted Transport

All traffic is served over HTTPS. Secrets and keys live in server-side configuration, never in client code.

Defence in depth: a coach can never load another coach’s bookings, and a referee can never read a thread they aren’t part of — the database refuses the query, regardless of what the app asks for.

Security & Data Protection — Safeguarding
06
Safeguarding by Design

Referees on the platform may be aged 14 to 17. Protecting them is not left to good intentions — the rules are enforced in the system itself.

  • Minimum age 14. Under-14s cannot register; date of birth is captured and validated at sign-up.
  • Age-appropriate matching. Eligibility is calculated for the actual match date, so referees are only offered fixtures suitable for their age.
  • Parental consent for under-16s. The account is locked until a parent or guardian verifies consent via a one-click email.
  • The lock is in the database. It is set by a server-side trigger at account creation, so no sign-up path can bypass it.
  • No direct messaging for under-16s. In-app chat is disabled; coaches are directed to email a parent or guardian for match updates.
  • Credentials tracked. DBS clearance and safeguarding status are recorded and monitored by administrators.
ENFORCED IN CODE
Locked • Consent-gated • Age-matched
There is no code path that lets an unverified under-16 account take bookings or message adults. Safeguarding is the default state, not a setting someone has to remember to switch on.

Security & Data Protection — Payments
07
Payments, Handled Safely

Money moves through the platform when coaches top up a wallet, funds are held in escrow for a booking, and referees are paid out. Every step is designed so that money is never lost, double-charged or stranded.

Stripe Handles Cards

Card details are entered with Stripe, a PCI-DSS Level 1 provider. Full card numbers never reach our servers.

Escrow Protection

A coach’s payment is held in escrow and only released to the referee once the match is confirmed complete.

No Double Charges

Every payment instruction carries an idempotency key, so a retry can never charge or pay twice.

Safe Withdrawals

Referee payouts use an audited multi-step process, so funds can never be debited without the transfer completing.

Verified Webhooks

Payment events from Stripe are cryptographically signature-checked and de-duplicated before they are trusted.

Weekly Reconciliation

An automated job checks balances and flags any stuck funds for administrators every week.

The financial ledger is immutable and retained even after an account is deleted — protecting both parties and meeting UK money-movement and audit obligations.

Security & Data Protection — Deletion & Rights
08
Delete Your Data, Anytime

You can delete your account yourself from inside the app — Profile › Delete account — meeting Apple App Store and Google Play requirements. Deletion erases your personal data while preserving the financial records the law requires us to keep.

What we erase / anonymise

  • Name, phone and avatar
  • Postcode and approximate location
  • Date of birth
  • FA registration number
  • Sign-in is disabled and the email is released

What we retain (and why)

  • Transaction, payout and escrow records
  • Dispute history
  • Kept only as long as law, tax and audit duties require
  • Held against an anonymised profile — no longer a contactable person
Money-safe: deletion is blocked while you still hold a wallet balance, a pending withdrawal, funds in escrow, or an active booking — so no funds are stranded and no counterparty is left mid-deal. You resolve those first, then delete.
Your UK GDPR rights
Access • Rectification • Erasure • Portability • Object / restrict processing • Withdraw consent. Exercise any of them at support@whistleconnect.co.uk; you may also complain to the ICO (ico.org.uk).

Appendix — Technical Security Controls
A1
Technical Security Controls

A reviewer-oriented summary of the mechanisms behind the overview. Implementation detail is in the codebase and migration history.

Access control
Row-Level Security on application tables; admin gating via a server-side is_admin() check; per-user read/write policies.
Privileged functions
Money/booking logic runs in SECURITY DEFINER functions with a pinned search_path; anon and PUBLIC EXECUTE are revoked (re-swept in a dedicated migration).
Authentication
Supabase Auth, password-based, with JWT sessions in cookies; profile creation uses bounded retry to avoid races.
Payment integrity
Idempotency keys on every Stripe write; atomic 3-step withdrawal with an audit table; escrow hold/release via RPC; webhook events de-duplicated; separate account- and Connect-scope signing secrets.
Account deletion
auth.uid()-scoped RPC (no id argument to abuse); anonymise-not-delete; pre-checks block deletion while value/commitments remain; wallet ledger is immutable by trigger.
FA verification
Single-use token links; processed by the service role only; data-minimised to three fields (name, FA number, county).
Safeguarding
Under-16 lock set in the handle_new_user database trigger — not bypassable from application code; messaging block enforced server-side.
Transport & secrets
HTTPS/TLS throughout; secrets in server-side environment configuration; push (VAPID) keys are not committed to source control.

Appendix — Continuous Hardening
A2
Continuous Hardening

We run automated Supabase security advisors and Sentry monitoring on an ongoing basis. The items below are known, tracked and mitigated, with planned hardening — shared here in the interest of transparency.

1   Direct execution of privileged database functions

Observation
Some privileged functions are callable by any signed-in user directly, not only through our server actions.
Current mitigation
Row-Level Security still applies; money functions are idempotent and re-validate their inputs, so direct calls cannot bypass ownership or double-charge.
Planned hardening
Per-function auth.uid() guards / move non-helper functions to a private schema.

2   PostGIS in the public schema

Observation
The geospatial extension and its static spatial-reference lookup table live in the public schema.
Current mitigation
The table holds only static reference data — no personal or sensitive information is exposed.
Planned hardening
Relocate PostGIS to a dedicated schema in a separately tested migration.

3   Leaked-password protection

Observation
The “have I been pwned” breached-password check is currently disabled.
Current mitigation
Standard Supabase Auth protections and password requirements remain in force.
Planned hardening
Enable breached-password screening, or raise the minimum password length.
